# PJ2 新增需求整理

来源：`pj2_descrip.md`

本文档整理 Project Evolution 2 中需要在现有 RISC 游戏基础上新增或调整的需求。重点包括 GUI、联网游戏、资源、地图属性、单位升级、战斗规则变化，以及课程交付物。

## 1. 非代码交付物

### 1.1 攻击指令 UI 原型

- 需要绘制“用户输入 attack order”的 UI 原型。
- 原型需要覆盖多个交互状态或多个屏幕。
- 原型必须展示用户如何选择：
  - 来源领地 `from territory`
  - 目标领地 `to territory`
  - 攻击单位数量 `number of units`
- 需要能在课堂展示中说明设计使用或计划使用的 UX 原则：
  - figure-ground
  - similarity
  - proximity
  - common region
  - continuity
  - closure
  - focal-point

### 1.2 团队会议录像

- 每个团队需要提交一次团队会议或 Scrum meeting 的录像。
- 录像中应讨论：
  - 当前进度
  - 问题与阻塞
  - 风险
  - 计划与任务管理
  - 版本管理
  - 测试

### 1.3 开发流程要求

- 继续使用 issues、feature branches、pull requests 和 code reviews。
- 代码需要保持注释合理、变量命名清晰、格式整洁、方法抽象良好。
- 团队应定义并遵守自己的 coding standard。
- 每位成员在 evolution 结束时需要完成个人贡献评估。

## 2. GUI 与客户端需求

- 游戏需要提供图形界面客户端，不能只保留文本界面。
- 图形界面形式不限，本项目可继续采用 Web 前端。
- 客户端需要清楚展示地图信息：
  - 哪些领地相邻或连通
  - 每个领地当前归属哪个玩家
  - 每个领地内各类型单位的数量
  - 每个领地产出哪些资源以及产出数量
  - 每个领地的 `size`
- 客户端可以提前校验用户输入，但服务器必须重新执行权威校验。
- 游戏结束时，客户端需要展示服务器宣布的胜利信息。
- 失败玩家应可以继续观看游戏，或选择断开连接。

## 3. 联网游戏需求

- 需要支持 2 到 5 名玩家从不同设备同时游玩。
- 系统需要区分服务器和图形客户端。
- 服务器负责执行并强制检查所有游戏规则。
- 服务器不强制要求持久化到进程退出之后，但可以选择实现持久化。
- 服务器必须支持同时存在多个游戏。
- 玩家需要有 login + password，使玩家可以回到自己的游戏。
- 一个玩家可以同时属于多个游戏。
- 客户端不要求同时显示多个游戏，但玩家应能在自己的活动游戏之间切换。

## 4. 地图与领地需求

- 地图仍然由多个领地构成。
- 每个领地任一时刻必须属于某一玩家。
- 每个领地包含单位数量。
- 每个领地必须与一个或多个其他领地相邻。
- 所有领地必须构成连通图，即任意领地都能到达任意其他领地。
- 地图布局细节由团队自行决定。
- 每个领地新增 `size` 属性：
  - `size` 表示移动穿过该领地的成本。
  - 初始领地分组中，各玩家起始领地的总 `size` 应保持等价。
- 每个领地需要有资源产出配置：
  - 可以产出一种或多种资源。
  - 每种资源的具体产量由团队自行决定。
  - 初始领地分组中，各玩家起始领地的资源产出总量应保持等价。

## 5. 初始化阶段需求

- 起始领地需要被划分为若干初始分组。
- 每个初始分组应包含相同数量的领地。
- 每位玩家选择或被分配一个初始领地分组。
- 每位玩家的起始领地组应有等价的资源产出。
- 每位玩家的起始领地组应有等价的总 `size`。
- 每位玩家获得相同数量的初始单位。
- 初始单位全部为 basic units。
- 玩家可以按自己的意愿把初始单位放置到自己的起始领地中。
- 初始单位数量可固定，也可作为新建游戏选项，但所有玩家必须相同。
- 初始单位放置需要同时进行：
  - 玩家放置并提交完成状态。
  - 在所有玩家完成之前，不能向其他玩家透露各自的放置情况。
- 所有玩家完成初始放置后，进入正常游戏流程。

## 6. 回合结构变化

每个回合仍然分三部分，但订单类型和资源消耗发生变化：

1. 发布订单 `Issue orders`
2. 执行订单 `Execute orders`
3. 接收新单位和资源 `Receive new units/resources`

### 6.1 订单类型

PJ2 中每回合支持三类订单：

- `upgrade`
- `move`
- `attack`

玩家每回合可以提交任意数量的 move、attack、upgrade 订单，但必须通过服务器合法性检查。

### 6.2 Move 订单

Move 订单必须指定：

- 移动单位数量
- 来源领地
- 目标领地

Move 合法性与效果：

- 来源与目标之间必须存在一条由该玩家控制的相邻领地组成的路径。
- Move 订单在 attack 订单之前生效。
- 移出来源领地的单位不参与来源领地防守。
- 移入目标领地的单位参与目标领地防守。
- Move 不得凭空创建单位，也不得让同一单位同时出现在多个地方。
- Move 订单新增 food 消耗：
  - 成本 = `移动路径经过领地的总 size * 移动单位数量`
  - 服务器需要选择合法路径中总成本最低的路径来计算消耗。

### 6.3 Attack 订单

Attack 订单必须指定：

- 攻击单位数量
- 来源领地
- 目标领地

Attack 合法性与效果：

- 目标领地必须由其他玩家控制。
- 目标领地必须与来源领地直接相邻。
- Attack 订单新增 food 消耗：
  - 每个攻击单位消耗 1 food。
- 所有攻击单位视为同时离开其来源领地，并同时到达攻击目标。
- 成功攻击会让攻击方获得目标领地所有权。
- 若 X 攻击 Y，且 Y 同时攻击 X，双方单位视为走不同路线，不会在途中交战。

### 6.4 提交与可见性

- 玩家可以在完成下单后提交订单。
- 玩家提交后，其订单视为 committed。
- 所有玩家提交之前，任何玩家不得看到其他玩家的订单。
- 所有玩家提交后，服务器统一执行订单。
- 已经失败的玩家不能再下订单，服务器应在每回合开始时自动把其订单视为空订单并提交。

### 6.5 回合结束奖励

- 每回合结束时，每个领地新增 1 个 basic unit。
- 玩家资源总量需要增加其回合结束时拥有领地的资源产出。
- 当回合中赢得新领地后，玩家在本回合结束时可以获得新领地的资源产出。

## 7. 资源需求

PJ2 至少需要支持以下资源概念：

- `food`
  - move 消耗 food。
  - attack 消耗 food。
- `technology`
  - 玩家最大科技等级升级消耗 technology。
  - 单位升级消耗 technology。

需要在玩家状态中记录资源总量，并在领地状态或领地定义中记录资源产出。

## 8. 战斗规则变化

服务器仍然负责解析每次 combat，并通知所有玩家结果。

### 8.1 单攻击方 vs 单防守方

- 战斗持续迭代，直到攻击方或防守方的参战单位耗尽。
- 每轮服务器为攻击方和防守方各掷一个 d20。
- 每个掷骰结果需要加上参与该轮战斗单位类型对应的 bonus。
- 点数较低的一方损失 1 个单位。
- 平局时防守方获胜，即攻击方损失 1 个单位。

### 8.2 单位配对顺序

单位 bonus 会影响战斗单位的配对顺序：

- 第一轮使用最高 bonus 攻击单位，对战最低 bonus 防守单位。
- 下一轮使用最低 bonus 攻击单位，对战最高 bonus 防守单位。
- 后续轮次在上述两种策略之间交替。
- 每轮损失单位后，下一轮基于剩余单位重新选择对应的最高或最低 bonus 单位。

### 8.3 多来源与多玩家攻击

- 同一玩家从多个领地攻击同一目标时，应合并为一个攻击部队。
- 多个玩家攻击同一目标时，服务器随机决定攻击方处理顺序。
- 第一场战斗的胜者会成为后续攻击的防守方。

## 9. 胜负需求

- 玩家不再控制任何领地时失败。
- 失败玩家不能再提交有效订单。
- 失败玩家可以继续观看游戏，或断开连接。
- 玩家控制地图全部领地时获胜。
- 玩家获胜后，服务器需要向所有剩余客户端宣布胜利。
- 游戏结束后，服务器可以退出，或提供开始新游戏的选项。

## 10. 科技等级与单位升级需求

### 10.1 玩家最大科技等级

- 每个玩家有 `maximum technology level`。
- 初始最大科技等级为 1。
- 玩家可以提交 upgrade order 来提升最大科技等级。
- 每回合最多只能提升 1 级。
- 最大科技等级升级需要 1 回合完成：
  - 本回合不可用。
  - 下一回合开始可用。

最大科技等级升级成本：

| 升级 | Technology 成本 |
| --- | ---: |
| 1 -> 2 | 50 |
| 2 -> 3 | 75 |
| 3 -> 4 | 125 |
| 4 -> 5 | 200 |
| 5 -> 6 | 300 |

### 10.2 单位类型与单位升级

- 玩家可以提交任意数量的单位升级订单。
- 单位升级消耗 technology resources。
- 单位升级会提高战斗 bonus。
- 成本按每个单位计算。
- 单位类型命名由团队自行决定，但 UI 和规则中必须清楚表达类型含义。
- 玩家可以在单个升级订单中把单位升级到任何允许的目标类型。
- 升级成本为目标单位累计成本与当前单位累计成本的差值。
- 玩家只能升级到其当前最大科技等级允许的单位类型。

单位升级表：

| 单位累计成本 | 升级增量成本 | Combat bonus | 所需科技等级 |
| ---: | ---: | ---: | --- |
| 0 | 0 | 0 | 初始 basic unit |
| 3 | 3 | 1 | 1 |
| 11 | 8 | 3 | 2 |
| 30 | 19 | 5 | 3 |
| 55 | 25 | 8 | 4 |
| 90 | 35 | 11 | 5 |
| 140 | 50 | 15 | 6 |

## 11. 实现清单建议

- 新增用户账户与登录密码机制。
- 新增多游戏管理，支持一个玩家加入多个游戏。
- 调整房间或游戏创建流程，支持 2 到 5 名真实联网玩家从不同设备进入同一局。
- 在地图模型中加入 territory size 和 resource production。
- 在玩家模型中加入 food、technology、maximum technology level，以及可能的 pending tech upgrade。
- 在单位模型中加入单位等级或单位类型，不再只用单一整数表示全部单位。
- 新增 upgrade 订单 DTO、校验逻辑和执行逻辑。
- 修改 move 校验与执行，加入 food 成本和最短成本路径计算。
- 修改 attack 校验与执行，加入 food 成本和单位类型。
- 修改 combat 逻辑，支持单位 bonus、最高/最低 bonus 交替配对、多来源合并和多玩家随机顺序。
- 修改回合结算，加入资源产出。
- 修改前端地图展示，显示资源产出、单位类型数量和 territory size。
- 修改前端下单 UI，支持 upgrade 订单、资源余额、food 消耗预览和 tech 消耗预览。
- 修改前端游戏列表或切换流程，使玩家可以返回或切换自己的活动游戏。
- 增加后端测试覆盖服务器权威校验、资源消耗、升级完成时机、最小移动成本路径、战斗 bonus 配对、多游戏与失败玩家自动提交。
- 增加前端测试或手动验收脚本覆盖攻击 UI、升级 UI、资源显示和游戏结束显示。
